設計を確定するために、当日その場で握りたい項目を 3階層で整理しました。

🔴 MUST（これを握らないと設計確定できない）

  • ①
    GA4 → BigQuery Export は既に有効化されているか→ OFFなら本日有効化＋過去データの backfill 方針（カラーミーCSV 取り込み等）の選択
  • ②
    GA4 の purchase イベントの送信状況・transaction_id の付与→ LTV分析の前提。届いていないと購買データが BigQuery で結合できない
  • ③
    utm_campaign の命名規則と既存キャンペーン一覧→ コホート分けの単位を決める。命名揺れがあれば正規化ロジックが必要
  • ④
    顧客ID紐付け方針（クッキー単位 / user_id送信実装 / メール突合）→ LTV分析の精度を決定。クッキー単位は最短だが端末変更で精度低下
  • ⑤
    コホート定義の合意：「広告経由初購入」の判定軸（初訪問 vs 初購入直前）・direct流入の扱い→ 後で揉めやすい論点。当日書面化したい
  • ⑥
    LTV期間定義：30日／60日／90日／1年 のどれを主軸に置くか→ ダッシュボードの設計が変わる
  • ⑦
    カラーミー売上の抽出経路（手動CSV / GAS / API）と更新タイミング→ 自動同期の頻度と仕組みが決まる
  • ⑧
    スプシのタブ構成・各タブの行数→ 読み取り制限の真因を確定し、BQ集約の集計粒度を決める

🟠 SHOULD（あれば設計精度が大きく上がる）

  • ⑨
    プロモコード／紹介コードの発行有無と紐付けデータの所在→ utm_campaign以外の経由判定軸が増え、LTV精度が上がる
  • ⑩
    Google広告アカウント構造（MCC配下か単独か）→ BQ Data Transfer の設定単位が変わる
  • ⑪
    Meta広告で利用している機能（Conversion API・CAPI実装の有無）→ iOS14以降のCV計測精度の議論に直結
  • ⑫
    既存 Looker Studio レポートのオーナー権限・データソース構成→ 新基盤を並列追加するか差し替えるかが決まる
  • ⑬
    「Aグループ」の定義（具体的にどの商品群か）→ 売上分析の確定例として活用

🔵 CONFIRM（権限・コスト面の確認事項）

  • ⑭
    GCP（BigQuery）プロジェクトは新規作成か既存利用か・課金口座の所在→ BQ コストはクライアント様負担となります（規模実測後に試算ご提示）
  • ⑮
    iOS14 ATT による Meta広告 CV計測の精度低下を許容するか→ GA4／BQ ファーストパーティ計測でカバー可能だが、限界は事前に共有
  • ⑯
    顧客ID紐付けで「メール突合パターン」を選ぶ場合の同意取得・個人情報保護法対応→ 会員規約の改定が必要になる可能性あり
既知の落とし穴 12点（同業案件で潰してきた知見）
「あとから揉める典型ポイント」を事前共有します。実装時はこれらを全て先回りで対策します。
①期間集計で率／ユニーク値が壊れる→ 合計同士の割り算（ratio of sums）で計算
②新規ユーザー率が100%超え→ user 単位の指標統一
③エンゲージメント率の異常スコア→ SUM／SUM の計算フィールドに統一
④滞在時間が ≒0 表示→ GA4 拡張計測の設定確認・必要時は追加実装
⑤％と小数の単位混在→ 全レイヤ小数で保持・最後だけ×100
⑥GA4 イベント未送信（page_view は届くがカスタムが届かない）→ BigQuery への着弾を必ず実検証
⑦Meta広告はネイティブ連携なし→ n8n 連携の工数差を見積で明示
⑧GA4→BQ Export は「設定後の日付からのみ」蓄積→ 過去データは別経路で backfill する選択肢を用意
⑨コホート定義の解釈ズレ→ 当日書面化（成功条件と紐付け）
⑩iOS14 以降の Meta 広告計測精度低下→ GA4／BQ のファーストパーティ計測でカバー
⑪クッキー単位 LTV の端末変更による分断→ ログイン時 user_id 送信実装を将来オプションに
⑫メール突合時の個人情報保護法対応→ 会員規約改定・同意取得文の追加
