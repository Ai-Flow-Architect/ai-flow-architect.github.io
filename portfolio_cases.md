### 事例1: AI求人応募自動化システム

**課題**
毎日3プラットフォーム（CrowdWorks・Coconala・Lancers）の案件を手動確認し、提案文を書いて応募する作業に3時間かかっていた。検索・スコアリング・文章作成・送信がすべて手動。

**解決策**
Python + Playwright + OpenAI API (GPT-4o) + cron scheduling による完全自動化パイプラインを構築。
- 毎朝8:30に自動起動（cron）
- 各プラットフォームから最新案件を取得
- GPT-4oでスキルマッチ度をスコアリング（0〜100点）
- 閾値以上の案件に対して個別提案文を自動生成
- Playwright経由で自動送信
- 結果をTelegram + Lark通知でレポート

**効果**
- 作業時間: 3時間/日 → 10分以下（95%削減）
- 応募件数: 3〜5件/日 → 20件以上/日（4倍以上）
- 3プラットフォーム合計20件以上を毎日無人実行

**技術スタック**: Python, Playwright, OpenAI API, Make.com, Lark API, cron

### 事例2: マルチプラットフォーム問い合わせ自動返信

**課題**
3つのフリーランスプラットフォームへの問い合わせ・チャット返信が遅延し、案件獲得機会を逃していた。特に夜間・外出中は数時間返信できないケースがあった。

**解決策**
Playwright + GPT + Lark APIによる監視・分類・返信自動化システムを構築。
- cron 30分間隔で各プラットフォームを巡回監視
- 未読メッセージをGPTで分類（FAQ / 価格交渉 / 詳細確認 / スパム）
- FAQ・詳細確認カテゴリは即座に自動返信
- 価格交渉・判断要の案件はLark通知でエスカレーション
- 全ログをLark Bitableに自動記録

**効果**
- 返信時間: 平均数時間 → 5分以内（95%以上短縮）
- 24/7無人稼働（夜間・外出中も対応可能）
- 問い合わせ対応による機会損失をほぼゼロに削減

**技術スタック**: Python, Playwright, OpenAI API, Lark API, cron scheduling

### 事例3: CRM統合ダッシュボード（Lark Bitable）

**課題**
顧客情報はLarkメモ、案件進捗は頭の中、KPIはGoogleスプレッドシートと分散。全体像の把握に毎日30分かかり、フォローアップ漏れも発生していた。

**解決策**
Lark BitableをCRMハブとして4テーブル構造を設計・構築。
- 顧客テーブル: 基本情報・対応履歴
- 案件テーブル: ステージ管理（新規→返信→商談→受注→完了）
- アクションログ: 対応記録・フォローアップ予定
- KPIテーブル: 月次目標・実績・達成率を自動集計
- Pythonスクリプト（crm_query.py）でLark API経由の一括操作に対応

**効果**
- 手動集計作業 30分/日 を完全撤廃
- リアルタイムKPI可視化（達成率・案件数・対応遅延を即確認）
- フォローアップ漏れゼロ（ステージ管理で自動リマインド）

**技術スタック**: Python, Lark API (Bitable), cron, make.com

### 事例4: LP問い合わせフォーム GAS連携自動化

**課題**
AIFLOWのLPからの問い合わせを受信後、返信メール・Lark通知・CRM登録の3つを毎回手動で行っており、対応漏れと時間ロスが発生していた。

**解決策**
Google Apps Script（GAS）をトリガーハブとして、フォーム送信と同時に3つのアクションを自動実行。
- GASのonFormSubmit → LarkAPIに問い合わせ内容をPOST
- 同トリガーで送信者へ自動返信メール（Gmail経由）
- CRM（Lark Bitable）に顧客レコードを自動登録（ステージ「新規」）
- GitHub Actions deployと組み合わせ、本番LP（GitHub Pages）に自動適用

**効果**
- フォーム送信から3秒以内にLark通知・CRM登録が完了
- 顧客対応漏れゼロ・初動返信の自動化で信頼感向上
- 将来的なGPT自動返信拡張の基盤を構築済み

**技術スタック**: Google Apps Script, Lark API, Gmail API, GitHub Actions, Python

### 事例5: メルカリ再出品自動化（Playwright RPA）

**課題**
メルカリ出品商品のアクセス数維持のために定期的な再出品が必要だが、商品ごとに手動で取り下げ→写真再アップロード→情報入力→出品と繰り返す作業が発生していた。

**解決策**
PlaywrightによるブラウザRPAで全フローをスクリプト化。
- 出品中商品データの自動取得・構造化
- 商品画像の自動ダウンロード・ローカル保存
- 既存出品の自動取り下げ
- 画像・情報を使った新規出品の自動送信
- セッション管理・ログイン状態の自動維持

**効果**
- 手動操作ゼロ（全ステップ完全自動化）
- 商品数に関わらず10〜15分で処理完了（従来は商品数×5〜10分）
- 定期cron設定で定期再出品サイクルを無人化

**技術スタック**: Python, Playwright, cron, Lark API（完了通知）
